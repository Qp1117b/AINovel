## 1. 配置文件整体结构

一个完整的 JSON 配置文件应包含以下顶级字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `version` | string | 是 | 配置文件版本，建议使用语义化版本，如 `"1.0"`。 |
| `description` | string | 是 | 对配置文件的简要描述，如“完整 16 Agent 配置文件（含生图师）”。 |
| `mode` | string | 是 | 配置模式：`"normal"`（默认）、`"datafication"`（数据化模式）、`"interactive"`（互动小说创作模式）。 |
| `maxStateBooks` | number | 是 | 状态书的最大数量，必须为正整数。 |
| `stateTypeLimit` | number | 是 | 每本状态书允许的最大状态条目数。 |
| `maxImagesPerBook` | number | 否 | 每本图库书（`状态书-图库N`）允许的最大图片条目数，默认同 `stateTypeLimit`。 |
| `maxAudiosPerBook` | number | 否 | 每本音频库书（`状态书-音频库N`）允许的最大音频条目数，默认同 `maxImagesPerBook`。 |
| `maxConsecutiveReflows` | number | 是 | 同一源连续触发同一目标的最大次数，超过则本章废章。 |
| `maxReflowDepth` | number | 是 | 全局回流处理的最大迭代次数，防止无限循环。 |
| `apiConfigs` | object | 是 | 定义可用的 API 后端，键为配置 ID，值为具体配置对象。 |
| `workflowStages` | array | 是 | 工作流阶段数组，每个阶段定义执行顺序和包含的 Agent。 |
| `agents` | object | 是 | 所有 Agent 的定义，键为 Agent 唯一标识，值为 Agent 配置对象。 |
| `categories` | object | 是 | 预选分类定义，用于树形菜单选择。可为空对象 `{}`。 |
| `categoryGroups` | array | 是 | 分类互斥组定义，可为空数组 `[]`。 |
| `extensions` | object | 是 | 保留给用户自定义数据，系统不解析，可为空对象 `{}`。 |

## 2. 元信息

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `version` | string | 是 | 配置文件版本，建议使用语义化版本，如 `"1.0"`。 |
| `description` | string | 是 | 对配置文件的简要描述，如 “完整 16 Agent 配置文件（含生图师）”。 |
| `mode` | string | 是 | 配置模式：`"normal"`（默认）、`"datafication"`（数据化模式）、`"interactive"`（互动小说创作模式）。 |

**示例**：
```json
{
    "version": "1.0",
    "description": "完整 16 Agent 配置文件（含生图师）",
    "mode": "normal"
}
```

## 3. 系统限制与回流控制

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `maxStateBooks` | number | 是 | 状态书的最大数量，必须为正整数。系统会创建 `状态书-1` 到 `状态书-N` 的书本。 |
| `stateTypeLimit` | number | 是 | 每本状态书允许的最大状态条目数。优化师会据此判断是否可新增条目。 |
| `maxImagesPerBook` | number | 否 | 每本图库书（`状态书-图库N`）允许的最大图片条目数，若未提供则默认为 `stateTypeLimit`。 |
| `maxAudiosPerBook` | number | 否 | 每本音频库书（`状态书-音频库N`）允许的最大音频条目数，若未提供则默认使用 `maxImagesPerBook` 的值。 |
| `maxConsecutiveReflows` | number | 是 | 同一源连续触发同一目标的最大次数，超过则本章废章。 |
| `maxReflowDepth` | number | 是 | 全局回流处理的最大迭代次数，防止无限循环。 |

**示例**：
```json
{
    "maxStateBooks": 10,
    "stateTypeLimit": 100,
    "maxImagesPerBook": 50,
    "maxConsecutiveReflows": 3,
    "maxReflowDepth": 100
}
```

## 4. 工作流阶段定义

定义各阶段的执行顺序、包含的 Agent 及执行模式。**每个阶段对象必须包含以下所有字段**。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `stage` | number | 是 | 阶段序号，必须为正整数且递增。系统根据此序号决定执行顺序。 |
| `id` | string | 是 | 阶段唯一标识，用于 Agent 的 `layer` 字段引用以及 `inputs` 中的阶段引用。必须唯一，推荐英文小写。 |
| `name` | string | 是 | 阶段显示名称，如 `"策划层"`。 |
| `color` | string | 是 | 十六进制颜色码，如 `"#9b59b6"`，用于画布节点显示。 |
| `description` | string | 是 | 阶段描述，鼠标悬停时显示。可为空字符串。 |
| `agents` | array | 是 | 该阶段可能包含的所有 Agent 键的列表。实际执行时只调用用户勾选的。 |
| `mode` | string | 是 | 执行模式：`"serial"`（串行）或 `"parallel"`（并行）。阶段内所有 Agent 的 `parallel` 必须与此一致。 |

**重要规则**：系统在校验配置中强制要求阶段内所有 Agent 的 `parallel` 值与阶段 `mode` 严格匹配。混用将导致加载失败。

**示例**：
```json
"workflowStages": [
    {
        "stage": 1,
        "id": "planning",
        "name": "策划层",
        "color": "#9b59b6",
        "description": "负责整体剧情策划",
        "agents": ["P1", "P2", "P3"],
        "mode": "serial"
    },
    {
        "stage": 2,
        "id": "design",
        "name": "设计层",
        "color": "#3498db",
        "description": "负责具体设计",
        "agents": ["D1", "D2", "D3"],
        "mode": "serial"
    },
    {
        "stage": 3,
        "id": "quality",
        "name": "审查层",
        "color": "#e74c3c",
        "description": "负责各项审查",
        "agents": ["Q1", "Q2", "Q3"],
        "mode": "parallel"
    }
]
```

## 5. API 配置

在根对象中添加 `apiConfigs` 字段，定义可用的 API 后端。每个配置项是一个对象，键为配置 ID（如 `"deepseekFast"`），值包含以下字段。

### 5.1 公共字段

所有类型（文本、图像、音频）的配置都必须包含以下字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `type` | string | 是 | `"text"`（文本生成）、`"image"`（图像生成）或 `"audio"`（音频生成）。 |
| `source` | string | 是 | 平台类型，必须与系统硬编码值完全一致（见 5.5 节枚举）。 |
| `apiUrl` | string | 是 | API 端点 URL，必须以 `http://` 或 `https://` 开头。 |
| `key` | string | 是 | API 密钥，**不能包含中文或非 ASCII 字符**。对于 SD WebUI 可留空。 |
| `model` | string | 是 | 模型名称。 |
| `timeout` | number | 否 | 请求超时时间（毫秒），默认 `60000`（文本）或 `120000`（图像）。 |

### 5.2 文本平台专用字段

当 `type: "text"` 时，可包含以下可选字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `maxTokens` | number | 最大生成 token 数，默认 `4000`。 |
| `temperature` | number | 温度参数，默认 `0.8`。 |
| `top_p` | number | 核采样参数。 |
| `frequency_penalty` | number | 频率惩罚。 |
| `presence_penalty` | number | 存在惩罚。 |
| `stop` | string/array | 停止序列。 |
| `logit_bias` | object | 词汇偏向（仅 OpenAI 兼容）。 |
| `n` | number | 生成几个候选，默认 `1`。 |
| `stream` | boolean | 是否流式输出，默认 `false`。 |
| `response_format` | object | 响应格式，如 `{ "type": "json_object" }`。 |

### 5.3 图像平台专用字段

当 `type: "image"` 时，必须包含 `mode` 字段，并可包含以下可选字段：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `mode` | string | 是 | 图像模式：`"txt2img"`（文生图）、`"img2img"`（图生图）、`"fusion"`（融合图）。 |
| `size` | string | 否 | 图像尺寸，如 `"1024x1024"`。 |
| `steps` | number | 否 | 采样步数，默认 `30`。 |
| `cfg_scale` | number | 否 | 提示词引导比例，默认 `7`。 |
| `quality` | string | 否 | 生成质量（DALL·E 3 支持 `"standard"` 或 `"hd"`）。 |
| `samples` | number | 否 | 一次生成的图片数量，默认 `1`。 |
| `negative_prompt` | string | 否 | 反向提示词。 |
| `sampler_name` | string | 否 | 采样器名称。 |
| `width` / `height` | number | 否 | 图片宽度和高度。 |
| `seed` | number | 否 | 随机种子，-1 表示随机。 |
| `batch_size` | number | 否 | 批次大小。 |
| `restore_faces` | boolean | 否 | 是否启用面部修复。 |
| `tiling` | boolean | 否 | 是否生成无缝平铺图案。 |

### 5.4 音频平台专用字段

当 `type: "audio"` 时，必须包含 `mode` 字段，并可包含其他平台特有字段（如 `voiceId`、`duration` 等）：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `mode` | string | 是 | 音频模式：`"music-generation"`（音乐生成）、`"voice-cloning"`（语音克隆）、`"audio-editing"`（音频编辑）。 |

### 5.5 平台 source 枚举

所有值必须为全小写字符串。

**文本平台**：
```
openai, claude, gemini, wenxin, qwen, glm, mistral, siliconflow, huggingface, groq, inference, openrouter, 4sapi, other
```

**图像平台**：
```
openai, stability, midjourney, sora, flux, picsart, siliconflow, sdwebui, other
```

**音频平台**：
```
elevenlabs, qwen, minimax, minimax-music, minimax-speech, mureka, mubert, lalal, huggingface, aiva, wondera, openai-tts, edge-tts, azure-tts, google-tts, stableaudio, riffusion, audiocraft, custom, local, other
```

### 5.6 配置示例

**OpenAI DALL·E 3**（文生图）
```json
"dalle3": {
    "type": "image",
    "mode": "txt2img",
    "source": "openai",
    "apiUrl": "https://api.openai.com/v1",
    "key": "sk-...",
    "model": "dall-e-3",
    "size": "1024x1024",
    "n": 1,
    "quality": "standard",
    "response_format": "url",
    "timeout": 60000
}
```

**本地部署 Stable Diffusion WebUI**（文生图）
```json
"sdWebUI": {
    "type": "image",
    "mode": "txt2img",
    "source": "sdwebui",
    "apiUrl": "http://localhost:7860",
    "key": "",
    "model": "v1-5-pruned-emaonly.safetensors",
    "steps": 20,
    "cfg_scale": 7,
    "width": 512,
    "height": 512,
    "seed": -1,
    "sampler_name": "DPM++ 2M Karras",
    "batch_size": 1,
    "restore_faces": false,
    "tiling": false,
    "negative_prompt": "nsfw, low quality, blurry",
    "timeout": 120000
}
```

**重要约束**：
- 整个配置文件中 **只能有一个 `type: 'image'` 的配置**，且必须包含 `mode` 字段。
- 文本 Agent（包括 `role: "imageGenerator"`）的 `apiConfigId` 必须指向 `type: 'text'` 的配置，或留空。
- 图像生成 Agent（`role: "imageGenerator"`）**不应设置 `apiConfigId`**，系统会自动使用唯一的图像 API。
- 音频 Agent 的 `apiConfigId` 必须指向 `type: 'audio'` 的配置，且 `mode` 必须匹配 Agent 的 `role`。
- 同一 `mode` 的音频配置只能有一个（如 `music-generation` 只能出现一次）。

## 6. Agent 定义

每个 Agent 以唯一键（如 `"P1"`）标识，值是一个包含以下字段的对象。**所有字段均为必填**（除非特别说明）。

### 6.1 基础信息

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 所关联的 SillyTavern 角色卡名称，必须与导入的角色卡完全一致。 |
| `displayName` | string | 在 UI 中显示的名称，用于区分复用同一角色卡的 Agent。 |
| `hover` | string | 鼠标悬停提示，简要说明职责。 |
| `layer` | string | 所属阶段 ID，必须与 `workflowStages` 中的某个 `id` 匹配。 |
| `order` | number | 同一阶段内的执行顺序，数值越小越先执行。 |
| `required` | boolean | 是否为必需 Agent。若为 `true`，则始终加入启用列表，用户不可取消。 |
| `parallel` | boolean | 是否可并行执行。`true` 表示可与其他并行 Agent 同时执行，且必须放在 `mode: "parallel"` 的阶段；`false` 必须放在 `mode: "serial"` 的阶段。 |
| `apiConfigId` | string | 引用自定义 API 配置的 ID。若为空字符串 `""`，则使用 SillyTavern 当前全局后端。**生图师（`imageGenerator`）必须留空**，系统自动使用唯一的图像 API 配置。 |
| `review` | boolean | 是否开启人工审核。若为 `true`，则该 Agent 的输出会弹出审核模态框，用户可选择“继续”或“打回”。打回时会触发回流。 |

### 6.2 输入相关字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `inputs` | array | 依赖的输入来源。每个元素可以是：`"user"`、`"auto"`、`"before"`、阶段 ID、其他 Agent 键、Agent 键 + `.last` 后缀、`"AgentKey.raw"`（获取原始输出）、`"read.xxx"`（读取文件）、`"save.xxx"`（保存文件）。 |
| `inputTemplate` | string | 输入提示词模板。使用 `【】` 作为占位符，数量必须与 `inputs` 长度相等。 |
| `inputMode` | array | 输入模式数组，长度与 `inputs` 相等。可选值：`"txt"`、`"status"`、`"chapter"`、`"all"`。对于非 `"user"`/`"auto"`/文件源的源，必须设为 `"txt"`。当 `inputs` 对应项为 `"auto"` 时，该值决定提取内容类型。 |
| `autoConfig` | array | 自动提取配置数组，长度与 `inputs` 相等。仅当对应源为 `"auto"` 时生效，值为提取章节数量（0=全部，N>0=最近N章）。非 `"auto"` 源必须设为 `0`。 |
| `inputPrompts` | array | 输入提示词数组，长度与 `inputs` 相等。当对应源内容为空触发用户输入时，在进度区域显示的提示文字。不需要用户输入的源留空字符串 `""`。 |

**文件源说明**：
- `"read.png"`：用户需上传一张 PNG 图片，系统返回该图片的 Data URL。
- `"read.txt"`、`"read.html"`、`"read.js"`：用户上传对应文本文件，系统返回文件内容。
- `"save.png"`：用户上传图片，系统保存到 `ImageStore` 并返回图片 ID。
- `"save.txt"`、`"save.html"`、`"save.js"`：用户上传文本文件，系统保存到 `OtherFileStore` 并返回文本 ID。
- `"save.audio"`：用户上传音频文件，系统保存到 `AudioStore` 并返回音频 ID。

### 6.3 控制流字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `reflowConditions` | array | 回流触发条件数组。当输出中包含任意字符串时触发回流。若无条件，设为空数组 `[]`。 |
| `executeInterval` | number | 周期性执行间隔。设为大于0的整数N，则仅在章节号能被N整除时执行；设为0则每章执行。 |

### 6.4 角色与描述字段

| 字段            | 类型     | 说明                                                                      |
| ------------- | ------ | ----------------------------------------------------------------------- |
| `outputs`     | array  | 输出内容描述数组，仅用于 UI 提示，不参与逻辑。例如 `["章节定位", "硬指标约束"]`。即使没有具体描述，也必须设为空数组 `[]`。 |
| `description` | string | 功能描述，显示在 UI 中作为提示信息。必须提供，可为空字符串。                                        |
| `role`        | string | 特殊角色标识，用于在脚本中动态查找关键 Agent。可选值详见第7节。可选角色除了 `saver `外若存在则必须唯一。            |

### 6.5 阶段输出合并模板

当 `inputs` 中包含阶段 ID 时，系统会收集该阶段所有已执行 Agent 的输出，并按以下格式合并：

```
Agent名称1：
Agent输出1

Agent名称2：
Agent输出2

...
```

如果该阶段没有已执行的 Agent，则合并结果为空字符串。

### 6.6 元 Agent 规范

元 Agent 指多个 Agent 共享同一张 SillyTavern 角色卡（`name` 字段相同）的设计模式。

**配置要求**：
- `name` 字段必须完全相同。
- `displayName` 必须不同，用于 UI 区分。
- `inputTemplate` 必须包含明确的角色描述和输出格式要求。

**元 Agent 输入模板严格规范**：
```
【职能】你现在是【】，请【】
【职能规则】1.【】；2.【】
【唯一输出格式】===标题===
【内容】...
【输出示例】===标题===
示例内容
【自定义】...
【输入】【】【】
```

### 6.7 Agent 配置最佳实践

- **依赖选择**：尽量使用具体 Agent 键而非阶段 ID，以减小回流范围。
- **跨章记忆**：使用 **Agent 键 + `.last`** 后缀（如 `"S1.last"`），必须使用 Agent 键，不能使用角色名。
- **输入模板**：`【】` 数量与 `inputs` 严格相等，并显式添加换行符控制格式。
- **自动提取**：`autoConfig:0` 提取所有历史章节，注意 token 消耗。
- **回流条件**：关键词应具体，避免过于宽泛。
- **周期性执行**：确保被跳过的 Agent 不被其他 Agent 依赖。
- **输入提示词**：为可能触发用户输入的源设置清晰提示词。
- **阶段与并行属性**：确保 `parallel` 值与所在阶段 `mode` 严格匹配。
- **自定义 API**：合理配置 `apiConfigId`，利用不同模型优化成本和质量。
- **人工审核**：为关键输出开启 `review` 字段，便于人工干预。

## 7. 角色规范（强制角色）

以下角色的输出格式被系统强制解析和依赖，必须严格遵循其规范。**所有角色在整个配置文件中必须唯一**。

### 7.1 最终文章师（finalChapter）

**职责**：根据所有前置信息生成完整的章节文本，包括章节标题、正文、结尾状态段落及原始状态草稿。

**输出格式要求**：
- 第一行必须是 `第X章 标题`（X 为章节号），系统将据此提取标题。
- 正文后可以包含任何内容，但建议最后包含 `===原始状态草稿===` 部分（可选，供其他 Agent 参考）。

### 7.2 剧情概览师（storySummarizer）

**职责**：每章结束后生成累积故事概览，通过 `.last` 自引用实现增量更新，反映整个故事从开篇到当前章节的完整发展脉络。

**输出格式**：自由格式，但建议使用结构化 Markdown，如：
```markdown
## 核心剧情
（主线进展）

## 人物变化
（人物关系、能力变化）

## 未解决悬念
（待揭晓的伏笔）

## 情感基调
（整体氛围）
```

### 7.3 状态优化师（optimizer）

**职责**：根据故事概览和系统提供的状态书当前快照，优化状态模板体系，输出格式化调整指令。

**输出格式**：
```
---需调整的状态模板---
书号-条目uid: **类别编号:类别名称**
- 字段名1: 占位符
- 字段名2: 占位符
...
属性行：字段1：值1；字段2：值2；...
```

- **头部**：固定为 `---需调整的状态模板---`，独占一行。
- **指令块**：每个块对应一个条目的修改/新增，块间用空行分隔。
- **删除操作**：仅允许在合并整合时出现，格式为 `书号-条目uid: delete`。
- **模板字段**：首行为 `**类别编号:类别名称**`，后续每行以 `- ` 开头定义字段，占位符只能用 `X`（数值型）或 `(选项/选项/...)`（枚举型），不得使用具体字母。
- **属性行**：紧接模板内容，可多行，用中文分号 `；` 分隔键值对。仅列出需修改的字段，未列出的保留原值。新增条目时需提供完整关键字段。

### 7.4 状态维护师（updater）

**职责**：根据最终章节内容和故事概览，为每本状态书生成续写锁定协议，更新状态条目中的具体数值。

**输出格式**：
```
===续写锁定协议===

**类别1:类别名称**
- 字段名1: 值1（变化说明）
- 字段名2: 值2（变化说明）
...

**类别2:类别名称**
...
```

- **协议头部**：固定为 `===续写锁定协议===`。
- **类别行**：`**类别编号:类别名称**`，必须与模板完全一致。
- **字段行**：`- 字段名: 值（变化说明）`，字段名与模板一致。
- **值**：必须为实际数值（如 `85/100`、`50%`），不得使用占位符。
- **变化说明**：用括号标注变化量及原因，如 `（-7，战斗受伤）`；若无变化则标注 `（无变化）`。

### 7.5 生图师（imageGenerator）

**职责**：根据章节内容生成图像提示词，调用图像 API 生成多张独立元素图片（背景、人物、物品等），并输出图片 ID 列表。

**输出格式**：
```
===图片生成指令===
段落X：
提示词内容

段落Y-Z：
提示词内容
...
```

- **头部**：固定为 `===图片生成指令===`，独占一行。
- **指令块**：每个块对应一个元素，以“段落X”或“段落X-Y”开头，后跟多行提示词。X 为章节段落序号（从 1 开始计数）。
- **提示词要求**：背景提示词需包含场景描述、光影方向、风格；人物、物品提示词需包含“透明背景”要求，如 `transparent background, isolated on white, PNG format`。
- **系统解析**：系统会为每个块调用图像 API，生成图片并保存到 `ImageStore`，然后在对应段落之后插入图片占位符 `![描述](id:xxx)`。

### 7.6 排版师（typesetter）

**职责**：根据最终章节内容、图片 ID 列表和当前查看器环境信息，生成最终的 HTML/CSS 排版代码。

**输出格式**：可以是纯 HTML，也可以是包含样式和脚本的完整页面。系统会原样保存。

### 7.7 融合生图师（fusionGenerator）

**职责**：将元素生图师生成的多个独立元素图片通过递进式 ControlNet 融合生成一张完整的场景图，输出融合图片 ID。

**输出格式**（JSON 对象）：
```json
{
  "preprocessSteps": [
    { "type": "depth", "sourceId": "bg123", "saveAs": "control_depth_bg" },
    { "type": "openpose", "sourceId": "char456", "saveAs": "control_pose_char" }
  ],
  "fusionSteps": [
    { "step": 1, "type": "depth", "prompt": "生成迷雾森林背景...", "sourceRef": "control_depth_bg", "controlId": "control_depth_bg", "denoise": 0.75 },
    { "step": 2, "type": "openpose", "prompt": "添加旅行者背影...", "sourceRef": 1, "controlId": "control_pose_char", "denoise": 0.65 }
  ]
}
```

### 7.8 图片管理员（imageLibrarian）

**职责**：根据本章新生成的图片列表、章节正文、故事概览和图库现状，生成图库更新指令（新增、修改、删除条目）。

**输出格式**（无分隔符版）：
```
新增：
字段1：值1；字段2：值2；...

修改：
书号-uid
字段1：值1；字段2：值2；...

删除：
书号-uid，书号-uid，...
```

### 7.9 变化生图师（imageVariator）

**职责**：根据本章章节正文和当前图库活跃摘要，从图库中选择需要修改的图片，生成变体（修改后）的新图片，输出 JSON 指令。

**输出格式**：
```json
[
  {
    "sourceId": "img_123",
    "prompt": "将服装改为红色，表情变为惊讶",
    "params": { "denoising_strength": 0.8 }
  }
]
```

### 7.10 音乐生成师（musicGenerator）

**职责**：根据输入生成音乐音频，输出音频 ID。

**输出格式**（JSON 数组）：
```json
[
  {
    "name": "战斗音乐",
    "range": [1,2],
    "id": "audio_xxx"
  }
]
```

### 7.11 语音克隆师（voiceCloner）

**职责**：根据音频样本和文本，克隆语音生成新音频，输出音频 ID。

**输出格式**：
```json
[
  {
    "name": "角色语音",
    "sample_id": "audio_yyy",
    "text": "你好，我是...",
    "id": "audio_zzz"
  }
]
```

### 7.12 音频编辑师（audioEditor）

**职责**：对源音频进行编辑（如分离、变调、混音），输出编辑后的音频 ID。

**输出格式**：
```json
[
  {
    "name": "编辑后音频",
    "source_id": "audio_xxx",
    "id": "audio_yyy"
  }
]
```

### 7.13 音频管理员（audioLibrarian）

**职责**：管理音频库条目，输出新增、修改、删除指令，格式与图片管理员类似。

### 7.14 交互分析助手（interactionAnalyzer）

**职责**：分析章节正文，识别可能的分支点，输出互动选项 JSON 数组。

**输出格式**：
```json
[
  { "key": "接受委托", "text": "接受委托", "target": 0 },
  { "key": "拒绝", "text": "拒绝", "target": 0 }
]
```

### 7.15 最终保存器（saver）

**职责**：在需要时生成最终保存内容，优先级高于 `finalChapter` 和 `typesetter`。

### 7.16 其他角色

其他未列出的 `role` 值可作为普通标识，系统不会强制解析，但可通过 `.last` 引用其输出。

## 8. 跨章记忆与状态持久化

### 8.1 跨章记忆机制（.last）

系统维护 `WORKFLOW_STATE.chapterMemory`，以 Agent 的 `role` 为键，存储上一章输出。

- 在 `inputs` 中引用 `.last` 时，必须使用 Agent 键（如 `"S1.last"`）。
- 若目标 Agent 未启用，视为缺失，触发用户输入并缓存（主流程）。
- 回流过程中，若记忆存在则直接使用，否则重新请求用户输入，不使用主流程缓存。
- 本章结束后，所有具有非空 `role` 的 Agent 输出更新到记忆中。

### 8.2 状态快照

每章开始前系统自动创建状态书快照，包含所有状态条目的名称和内容。快照用于：
- 废章回滚
- 章节保存时存入历史记录
- 用户回滚操作

### 8.3 多状态书支持

状态书命名规则：`状态书-N`（N 从 1 开始递增）。
- 状态模板条目：`状态模板-N`
- 状态条目：`状态-{书号}-{类别编号}-{类别名}`

维护师会遍历所有激活的状态书，为每本书生成协议。若任意一书更新失败，本章标记为废章并回滚。

## 9. 工作流执行原理

### 9.1 输入整合流程

当 Agent 的所有依赖就绪，系统收集 `inputs` 中每个源的内容：

- `"user"`：直接取用户输入（主流程缓存，回流不缓存）。
- 其他 Agent 键：取该 Agent 本章输出。若目标未启用，触发用户输入并缓存；若输出不存在，传递空字符串。
- `.last`：按跨章记忆规则处理。
- 阶段 ID：取该阶段所有已执行 Agent 的合并文本。若阶段内无输出，触发用户输入。
- `"auto"`：从历史章节提取内容，模式由 `inputMode` 决定。若历史章节为空，触发用户输入。
- `"before"`：取前一个串行 Agent 输出（或并行阶段快照）。若内容为空，触发用户输入。
- `"read.xxx"`：触发文件上传模态框，用户上传文件后返回内容或 Data URL。
- `"save.xxx"`：触发文件上传模态框，用户上传文件后保存到对应 Store 并返回 ID。

然后按顺序将每个 `【】` 占位符替换为对应内容，生成最终提示词。

### 9.2 回流机制

当 Agent 输出触发 `reflowConditions`：

1. **确定回流目标**：遍历其 `inputs`，收集所有非 `"user"` 的输入源。
2. **状态变更**：触发源设为 `reflow_waiting`，目标设为 `reflow_processing`，并清除旧输出。
3. **反馈准备**：将触发源的输出作为反馈存储到每个目标的反馈映射中。
4. **重新调度**：立即处理所有 `reflow_processing` 的 Agent（按 `order` 串行），其提示词末尾附加反馈。
5. **等待唤醒**：当目标完成，检查所有 `reflow_waiting` 的 Agent，若等待的目标全部完成，则将其设为 `pending` 并调度执行。
6. **循环控制**：系统记录每个目标被同一源连续触发的次数，若超过 `maxConsecutiveReflows`，则本章废章。

并行阶段中，回流触发后不会立即执行，而是等阶段结束后统一处理。

### 9.3 周期性执行

若 Agent 的 `executeInterval` > 0，系统在调用前检查当前章节号能否被该值整除。若不能，则跳过，输出为空字符串，状态设为 `completed`。

### 9.4 废章与回滚

若 `updater` 更新失败，`WORKFLOW_STATE.discarded` 设为 `true`。所有阶段完成后：
- 使用本章开始前的快照回滚所有状态书。
- 跳过章节保存。
- 废章内容保留在 `discardedChapter` 中供查看。

## 10. 预选分类定义

用户可在此选择不同的创作倾向，系统根据选项推荐的 Agent 列表自动勾选。**每个分类对象必须包含以下字段**。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 分类显示名称。 |
| `description` | string | 是 | 分类描述。 |
| `selectionMode` | string | 是 | 选择模式：`"single"`（单选）或 `"multiple"`（多选）。 |
| `options` | object | 是 | 选项集合，键为选项标识，值为选项对象。每个选项对象必须包含：`name` (string)、`description` (string)、`icon` (string)、`agents` (array)。 |

**示例**：
```json
"categories": {
    "qualityLayer": {
        "name": "质量分层",
        "description": "选择创作质量标准",
        "selectionMode": "single",
        "options": {
            "minimal": {
                "name": "极简模式",
                "description": "仅必要 Agent，最快速度",
                "icon": "⚡",
                "agents": ["P1", "W1", "M2"]
            },
            "quick": {
                "name": "快速模式",
                "description": "最小 Agent 集，追求速度",
                "icon": "🚀",
                "agents": ["P1", "D2", "W1", "M2"]
            }
        }
    }
}
```

定义哪些分类不能同时被选择（组内单选）。**每个组对象必须包含以下字段**。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 组名。 |
| `categories` | array | 是 | 组内分类的键列表，这些分类的选项互斥。可为空数组。 |

**示例**：
```json
"categoryGroups": [
    { "name": "主要模式组", "categories": ["qualityLayer", "novelType"] }
]
```

## 12. 配置校验要求

系统加载配置时会进行严格校验，必须满足以下条件：

1. **必需字段**：每个 Agent 必须包含所有必填字段。
2. **字段类型**：类型必须符合定义。
3. **输入源有效性**：`inputs` 中的元素必须是 `"user"`、`"auto"`、`"before"`、已定义的阶段 ID、已定义的 Agent 键、已定义的 Agent 键加 `.last` 后缀，或 `"read.xxx"`/`"save.xxx"` 格式。
4. **数组长度一致**：`inputMode`、`autoConfig`、`inputPrompts` 的长度必须与 `inputs` 相等。
5. **占位符数量**：`inputTemplate` 中的 `【】` 数量必须与 `inputs` 长度相等。
6. **回流条件类型**：`reflowConditions` 必须为字符串数组（可为空）。
7. **工作流阶段**：`workflowStages` 必须存在，`stage` 为正整数且递增，`mode` 为 `"serial"` 或 `"parallel"`。
8. **阶段与并行属性匹配**：每个阶段内所有 Agent 的 `parallel` 必须与 `mode` 一致。
9. **回流控制配置**：必须包含 `maxConsecutiveReflows` 和 `maxReflowDepth`，且为正整数。
10. **`apiConfigs` 存在性**：必须包含 `apiConfigs` 字段，且为对象（可为空）。
11. **`apiConfigs` 结构**：每个配置项必须包含 `type`、`source`、`apiUrl`、`key`、`model`，且值非空；`type` 必须为 `"text"`、`"image"` 或 `"audio"`；`source` 必须为对应枚举值；`apiUrl` 必须是有效 URL。
12. **图像配置唯一性**：整个配置文件中必须恰好有一个 `type: "image"` 的配置（当存在 `imageGenerator` 时强制，否则最多一个）。
13. **音频配置模式唯一性**：同一 `mode` 的音频配置只能有一个（如 `music-generation` 只能出现一次）。
14. **`apiConfigId` 引用有效性**：若 Agent 的 `apiConfigId` 非空，则必须在 `apiConfigs` 中存在对应 ID，且该配置的 `type` 与 Agent 的 `role` 匹配（文本 Agent 需 `type: "text"`，音频 Agent 需 `type: "audio"`）。
15. **角色唯一性**：以下角色在整个配置文件中只能出现一次：`finalChapter`、`optimizer`、`updater`、`storySummarizer`、`imageGenerator`、`typesetter`、`fusionGenerator`、`imageLibrarian`、`imageVariator`、`musicGenerator`、`voiceCloner`、`audioEditor`、`audioLibrarian`、`interactionAnalyzer`、`saver`。
16. **生图师专用配置**：若存在 `imageGenerator`，则必须存在一个 `type: "image"` 的配置，且该 Agent 的 `apiConfigId` 必须留空。
17. **阶段 ID 存在性**：每个 Agent 的 `layer` 字段值必须在 `workflowStages` 的 `id` 中存在。

## 14. 常见问题与注意事项

1. **Agent 名称必须与角色卡完全一致**：包括大小写和空格。
2. **`.last` 引用**：必须使用 Agent 键，不能使用角色名；访问受目标启用状态影响；回流中若记忆不存在会请求用户输入。
3. **`before` 引用**：若指向内容为空，会转为用户输入并缓存（主流程）；回流中重新请求。
4. **周期性执行**：被跳过的 Agent 不会产生输出，依赖它的其他 Agent 会因缺失而触发用户输入。
5. **状态书序号必须连续**：系统会检测到第一个不存在的状态书后停止。
6. **状态模板条目必须存在**：维护师需要读取模板，否则协议生成异常。
7. **废章回滚**：废章时回滚到本章开始前的状态快照，但已保存的章节不会删除。
8. **回流深度限制**：`maxReflowDepth` 控制最大迭代次数，超过则停止并报错。
9. **配置文件更新**：修改后需重新加载才能生效。
10. **避免循环依赖**：确保 `inputs` 不形成循环。
11. **输入提示词**：为可能触发用户输入的源设置清晰提示词。
12. **核心角色强制规范**：`optimizer`、`updater`等的输出格式必须严格遵循。
13. **元Agent复用**：确保角色卡能解析 `inputTemplate` 中的角色描述。
14. **阶段与并行属性严格匹配**：`parallel` 值与阶段 `mode` 必须一致。
15. **自定义 API 配置**：密钥不能包含中文或非 ASCII 字符。
16. **生图师使用**：输出格式必须包含 `===图片生成指令===`，且 `apiConfigId` 留空。
17. **排版师使用**：图片使用 `src="id:xxx"` 引用。
18. **图片存储**：图片保存到 IndexedDB，渲染时通过 `id:` 前缀加载。
19. **文件源**：`read.xxx` 和 `save.xxx` 会触发文件上传，确保 API 支持文件处理。
20. **人工审核**：开启 `review` 的 Agent 会在输出后弹出审核框，打回时将触发回流并提供用户反馈。

## 1. 需求分析与 Agent 规划

配置文件的核心是定义一组 Agent（即 SillyTavern 中的角色卡），每个 Agent 承担一项原子任务。

### 1.1 确定核心环节

根据源码中定义的 `role`，系统目前支持以下特殊角色，它们拥有专用的执行逻辑：

| 角色（role） | 职责 | 是否必须 |
|--------------|------|----------|
| `finalChapter` | 最终文章师，生成完整章节 | 否（但常用） |
| `optimizer` | 状态优化师，优化状态模板 | 否 |
| `updater` | 状态维护师，更新状态书 | 否 |
| `storySummarizer` | 剧情概览师，生成累积概览 | 否 |
| `imageGenerator` | 生图师，生成图像 | 否 |
| `fusionGenerator` | 融合生图师，融合多图 | 否 |
| `imageVariator` | 变化生图师，生成图片变体 | 否 |
| `imageLibrarian` | 图片管理员，管理图库 | 否 |
| `musicGenerator` | 音乐生成师，生成音乐 | 否 |
| `voiceCloner` | 语音克隆师，克隆语音 | 否 |
| `audioEditor` | 音频编辑师，编辑音频 | 否 |
| `audioLibrarian` | 音频管理员，管理音频库 | 否 |
| `interactiveAgent` | 互动师，处理用户交互 | 否 |
| `saver` | 最终保存器，保存章节 | 否 |
| `typesetter` | 排版师，生成最终 HTML | 否 |

> **注意**：`interactionAnalyzer` 虽在角色列表中存在，但**未在源码中实现**，请勿使用。其他未列出的 `role` 值可作为普通标识，通过 `.last` 引用其输出。

**核心常用角色**（建议配置）：
- `finalChapter`：生成完整的章节正文。
- `storySummarizer`：生成累积故事概览，用于跨章记忆。
- `optimizer` + `updater`：管理状态书，实现动态世界。
- `typesetter`：将章节内容排版为 HTML（支持图片嵌入）。

**可选角色**：图像、音频相关角色按需添加。

### 1.2 确定 Agent 数量与粒度

- **原子性**：每个 Agent 聚焦一个原子任务，输出应能被明确依赖。
- **独立性**：尽量让 Agent 之间依赖最小化，便于并行执行。
- **可组合性**：输出格式应易于被其他 Agent 组合使用。

**示例**：
- 若您不需要图像，则无需包含任何图像相关 Agent。
- 若需要复杂的状态管理，可配置多个 `optimizer` 和 `updater`（但注意角色唯一性约束，每个特殊角色只能出现一次）。

### 1.3 命名规范

使用简短、有意义的键名，推荐格式：`<环节标识><序号>`，例如：

- 策划：`P1`、`P2`
- 设计：`D1`、`D2`
- 审查：`Q1`、`Q2`
- 创作：`W1`（finalChapter）、`W2`
- 概览：`S1`（storySummarizer）
- 管理：`M1`（optimizer）、`M2`（updater）
- 图像：`G1`（imageGenerator）、`G2`（fusionGenerator）
- 音频：`A1`（musicGenerator）、`A2`（voiceCloner）
- 互动：`IA`（interactiveAgent）
- 保存：`SV`（saver）
- 排版：`TS`（typesetter）

## 2. 工作流阶段设计

工作流阶段（`workflowStages`）是配置的执行骨架。每个阶段代表一个执行步骤，包含该阶段内要执行的 Agent 列表及其执行模式。

### 2.1 阶段定义

每个阶段对象必须包含以下字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `stage` | number | 是 | 阶段序号，必须为正整数且递增。系统据此决定执行顺序。 |
| `id` | string | 是 | 阶段唯一标识，用于 Agent 的 `stage` 字段引用以及 `inputs` 中的阶段引用。必须唯一，推荐英文小写。 |
| `name` | string | 是 | 阶段显示名称，如 `"策划阶段"`。 |
| `color` | string | 是 | 十六进制颜色码，如 `"#9b59b6"`，用于画布节点显示。 |
| `description` | string | 是 | 阶段描述，鼠标悬停时显示。可为空字符串。 |
| `agents` | array | 是 | 该阶段可能包含的所有 Agent 键的列表。实际执行时只调用用户勾选的。 |
| `mode` | string | 是 | 执行模式：`"serial"`（串行）或 `"parallel"`（并行）。 |

**重要规则**：系统在校验配置时**不会**检查阶段内 Agent 的 `parallel` 属性（该属性已废弃），因此您无需在 Agent 中设置 `parallel` 字段。只需通过阶段 `mode` 控制执行方式即可。

### 2.2 阶段划分与并行/串行决策

- 根据依赖图将无依赖的任务分组，可并行的放入同一阶段，设置 `mode: "parallel"`。
- 有依赖关系的任务必须串行，放入不同阶段或同一串行阶段（按 `order` 顺序）。

**示例**：
- 并行阶段：三个审查员 `Q1`、`Q2`、`Q3`（无依赖）
- 串行阶段：创作层 `W1` 依赖审查层的汇总输出

### 2.3 聚合 Agent 的使用

当多个并行 Agent 的输出都需要被同一个下游 Agent 使用时，可以设计一个聚合 Agent，其输入为这些并行 Agent 的输出（通过具体键或阶段 ID），将信息整合后再传递给下游。聚合 Agent 必须放在一个**新的串行阶段**中。

**示例**：
- 并行阶段：三个审查员 `Q1`、`Q2`、`Q3`
- 串行阶段：聚合审查员 `Q_summary`，输入为 `["Q1", "Q2", "Q3"]`，输出一份综合报告给创作层

### 2.4 阶段颜色与视觉

阶段颜色仅用于画布显示，不影响执行逻辑。建议使用柔和的色调，并确保不同阶段颜色有明显区分。

## 3. 输入源设计方法论

`inputs` 数组定义了 Agent 的所有依赖。每个元素可以是以下几种类型，系统会根据类型动态获取内容。

### 3.1 输入源类型详解

| 类型 | 格式示例 | 说明 |
|------|----------|------|
| `"user"` | `"user"` | 用户原始输入（文本框内容）。每次遇到都会等待用户新输入，不缓存。 |
| 阶段 ID | `"planning"` | 依赖该阶段所有已执行 Agent 的合并输出。若阶段内无输出，转为用户输入。 |
| 具体 Agent 键 | `"P1"` | 依赖特定 Agent 在本章的实时输出。若该 Agent 未启用或输出为空，转为用户输入并缓存（主流程）；回流时不缓存，重新请求。 |
| `.last` 后缀 | `"S1.last"` | 跨章记忆，获取上一章该 Agent 的输出。首次执行时若记忆为空，转为用户输入并缓存（主流程）；回流时不使用缓存，重新请求。 |
| `"auto"` | `"auto"` | 从历史章节自动提取内容，需配合 `autoConfig` 和 `inputMode`。 |
| `"before"` | `"before"` | 在串行阶段内取前一个串行 Agent 的输出；在并行阶段内取进入阶段前的最后一个串行 Agent 快照。 |
| 文件源 `read.xxx` | `"read.png"` | 触发文件上传，用户上传图片后返回 Data URL。支持 `png`、`txt`、`html`、`js`。 |
| 文件源 `save.xxx` | `"save.png"` | 触发文件上传，用户上传图片后保存到 `ImageStore` 并返回 ID。支持 `png`、`txt`、`html`、`js`、`audio`。 |
| `id.xxx` | `"id.img_123"` | 直接引用已存储资源的 ID（图片、音频、文本），返回该资源的内容或 ID。 |

### 3.2 优先使用非具体 Agent 键

为最大限度减少用户输入，并提高配置灵活性，应优先考虑使用阶段 ID、`before`、`auto` 等非具体键，而非具体 Agent 键。原因：
- 避免因 Agent 未启用而触发用户输入。
- 适应预选变化，无需修改配置。
- 简化配置，更易读。

### 3.3 `before` 设计

- **串行阶段内**：`before` 指向前一个按 `order` 顺序执行的串行 Agent 的输出。
- **并行阶段内**：所有并行 Agent 的 `before` 都指向进入该并行阶段前的最后一个串行 Agent 的输出快照。
- **首个 Agent**：若工作流第一个 Agent 使用 `before`，则因无前驱而触发用户输入。

**设计建议**：在串行链中优先使用 `before` 替代具体 Agent 键，使配置简洁且适应预选变化。

### 3.4 阶段 ID 设计

- 阶段 ID 会合并该阶段所有已执行 Agent 的输出，格式为：
  ```
  Agent名称1：
  输出1

  Agent名称2：
  输出2

  ...
  ```
- 适用于需要整阶段信息的场景（如创作层需要设计阶段的全部意见）。
- **回流波及**：若依赖阶段 ID，回流时会重新执行整个阶段，请谨慎使用。

### 3.5 跨章记忆 `.last` 设计

- **必须使用 Agent 键**，如 `"S1.last"`，不能使用角色名。
- 首次执行时记忆为空，主流程中转为用户输入并缓存；同一章内后续相同 `.last` 复用缓存。
- 回流中，若记忆存在则直接使用，若不存在则转为等待用户输入（不使用主流程缓存）。
- 只有具有非空 `role` 的 Agent 的输出才会在章节结束时存入记忆。

### 3.6 `auto` 自动提取设计

- `autoConfig` 指定提取章节数（0=全部，N=最近 N 章）。
- `inputMode` 指定提取内容：`"status"`（状态快照）、`"chapter"`（正文）、`"all"`（两者）。
- 若历史章节为空，转为用户输入，模式保持原样（可能打开章节选择模态框）。

**优化**：提取全部章节可能消耗大量 Token，通常 N=3~5 足够提供上下文。

### 3.7 文件源 `read`/`save` 设计

- 适用于需要用户上传文件的场景，如上传参考图、音频样本等。
- `read.xxx` 返回文件内容或 Data URL，`save.xxx` 返回存储后的 ID。
- 需确保对应的 API 配置支持文件上传（如 OpenAI 的文件 API）。

### 3.8 `id.xxx` 直接引用

- 可直接引用已存储资源的 ID（如 `id.img_123`、`id.audio_456`、`id.other_789`）。
- 对于图片和音频，返回该资源的 ID 字符串（供后续使用）；对于文本，返回文件内容。

## 4. 输入模式与自动提取配置

`inputMode` 数组长度必须等于 `inputs`，每个元素对应相同位置输入源的输入方式：

- `"txt"`：普通文本输入（适用于大多数源）。
- `"status"` / `"chapter"` / `"all"`：仅当对应源为 `"auto"` 时生效，指定提取内容类型。

`autoConfig` 数组长度也必须等于 `inputs`，当源为 `"auto"` 时指定提取章节数，否则必须为 `0`。

**示例**：
```json
"inputs": ["auto", "user"],
"inputMode": ["all", "txt"],
"autoConfig": [5, 0]
```
表示第一个输入源自动提取最近 5 章的全部内容（正文+状态），第二个输入源为用户文本输入。

## 5. 输入模板编写技巧

- **使用 Markdown 结构**：用分隔符清晰分隔不同输入源，例如：
  ```markdown
  ### 设计层信息
  【】

  ### 上一章概览
  【】

  ### 历史参考
  【】

  请根据以上信息，生成本章设计。
  ```
- **处理空输入**：对可能为空的输入（如首次 `.last`），模板中应说明：“若无此信息，请忽略”。
- **强制输出格式**：对 `finalChapter`，可要求第一行格式 `第X章 标题`；对 `optimizer` 和 `updater`，要求严格遵循系统定义的协议格式。
- **引导用户输入**：对接收 `"user"` 的 Agent，可包含设定采纳引导，例如：“请分析用户需求，识别新设定，若冲突则输出‘需要用户澄清’”。

## 6. 回流机制设计

### 6.1 回流条件设计原则

- **精确性**：使用具有明确语义的短语，避免与正常输出中的常见词混淆。例如，使用“需要修改：”而非“修改”。
- **结构化**：建议在输出中为反馈内容设置固定的标记，如“【需要修改】”。
- **单义性**：每个回流条件应只对应一类问题。
- **条件数量**：每个 Agent 的回流条件不宜过多（建议≤3个）。

### 6.2 反馈内容规范化

当 Agent 决定触发回流时，其输出中应包含：
1. **回流标记**：即配置在 `reflowConditions` 中的关键词。
2. **结构化反馈**：紧随回流标记之后，用明确的格式描述需要修改的具体问题。

**推荐格式**：
```
【需要修改】
- 问题1：描述及建议修改方向
- 问题2：描述及建议修改方向
...
```

被回流 Agent 的提示词末尾会自动附加反馈，格式为：
```
【来自 XXX 的反馈】
反馈内容
```

### 6.3 并行阶段的回流处理

- 当回流发生在并行阶段时，不会立即执行，而是等该阶段所有 Agent 完成后统一处理。
- 若多个回流源同时触发且依赖同一目标，系统会合并依赖，并将所有反馈拼接后传递给目标 Agent。

## 7. 性能与 Token 开销优化

### 7.1 Token 消耗分析

总 token ≈ Σ（各 Agent 输入 + 输出）× (1 + 回流率)。输入通常为输出的 1~3 倍。

### 7.2 优化策略

| 策略 | 说明 |
|------|------|
| 精简输入模板 | 去除冗余文字，使用简洁标记。 |
| 使用聚合 Agent | 将多个长输出压缩成摘要。 |
| 控制自动提取章节数 | 通常 N=3~5 足够，避免提取全部。 |
| 明确输出长度 | 在角色卡中指定字数，如“请用500字内完成”。 |
| 周期性执行 | 将非每章必需的任务（如优化师）设为每5章执行一次。 |
| 精确回流 | 设置精准条件，避免误触发。 |
| 回流反馈精简 | 触发回流的 Agent 应输出简短的反馈，避免过长。 |
| 状态书模板优化 | 字段名使用简短词汇，如“修为”而非“人物修炼等级”。 |

## 8. 预选分类设计

### 8.1 分类维度选择

推荐至少两个维度：
- **质量分层**：控制 Agent 数量，如“极简模式”、“快速模式”、“精细模式”。
- **章节类型**：影响创作倾向，如“战斗章”、“对话章”、“探索章”。

### 8.2 选项设计

每个选项包含 `name`、`description`、`icon`、`agents`（推荐启用的 Agent 键列表）。章节类型选项的 `agents` 可为空数组，仅用于提示用户手动勾选。

### 8.3 互斥组的使用

若某些分类不能同时选中（如小说类型），用 `categoryGroups` 定义互斥组。组内分类的 `selectionMode` 必须为 `"single"`。

**示例**：
```json
"categoryGroups": [
    { "name": "主要模式组", "categories": ["qualityLayer", "novelType"] }
]
```

## 9. 生成配置文件的步骤

### 9.1 借助大模型生成初稿

1. **整理需求**：明确您的小说类型、复杂度、所需 Agent 数量、是否使用图像/音频等。
2. **提供示例或规则**：向大模型提供《配置文件编写指南》中的规范，并强调阶段必须纯串行/纯并行、角色唯一性等约束。
3. **生成初稿**：大模型会根据您的需求生成 JSON 配置文件。您可能需要多次调整输入模板中的占位符顺序、回流条件等。

### 9.2 手动校验与调整

1. **检查角色唯一性**：确保每个特殊角色（如 `finalChapter`、`optimizer` 等）最多出现一次。
2. **检查输入源有效性**：所有 `inputs` 中的元素必须有效，`.last` 后缀必须指向已存在的 Agent 键。
3. **检查占位符数量**：`inputTemplate` 中的 `【】` 数量必须等于 `inputs` 长度。
4. **检查 API 配置**：确保图像配置唯一，音频配置模式唯一。
5. **使用系统前置检测**：加载配置文件后，系统会自动进行校验，根据错误提示修正。

## 10. 示例：从零生成一个中配配置文件

**需求**：
- 中配修仙小说，无图像/音频需求。
- 策划1个（主线），设计1个（细节），审查1个（逻辑），创作1个（正文），概览1个，管理2个（优化师+维护师）。
- 用户输入只在第一章需要提供初始概览，后续自动运行。
- 优化师每5章执行一次。

**生成步骤**：

1. **确定 Agent 列表**：
   - `P1`：主线策划（无特殊 role）
   - `D1`：细节设计（无特殊 role）
   - `Q1`：逻辑审查（无特殊 role）
   - `W1`：最终章节创作师（`role: "finalChapter"`）
   - `S1`：剧情概览师（`role: "storySummarizer"`）
   - `M1`：状态优化师（`role: "optimizer"`）
   - `M2`：状态维护师（`role: "updater"`）

2. **设计依赖**：
   - `P1`：依赖 `"user"`（用户输入）
   - `D1`：依赖 `"before"`（前序策划输出）
   - `Q1`：依赖 `"before"`（前序设计输出）
   - `W1`：依赖 `"before"`（前序审查输出）和 `"S1.last"`（上一章概览）
   - `S1`：依赖 `"W1"` 和 `"S1.last"`
   - `M1`：依赖 `"S1.last"`，每5章执行
   - `M2`：依赖 `"S1.last"` 和 `"W1"`

3. **设计工作流阶段**：
   - 阶段1（策划阶段）：`["P1"]`，串行
   - 阶段2（设计阶段）：`["D1"]`，串行
   - 阶段3（审查阶段）：`["Q1"]`，串行
   - 阶段4（创作阶段）：`["W1"]`，串行
   - 阶段5（概览阶段）：`["S1"]`，串行
   - 阶段6（管理阶段）：`["M1", "M2"]`，**并行**（两者无依赖，可并行）

4. **编写 JSON**：根据以上设计，参考《配置文件编写指南》中的格式填写具体字段。

通过上述步骤，您可以生成一套满足需求、符合系统规范的配置文件。建议生成后使用系统前置检测功能进行验证，根据错误提示修正。
